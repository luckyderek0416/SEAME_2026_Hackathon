## 1.5 SW 코리도 추적 — 실차 검증 완료, 소스 기본 OFF (라이브로 켬)

하드코딩 호(시간 기반 개루프)가 배터리별 속도 차로 런마다 다른 호를 그리는 문제의
위치 기반 폐루프 대체. 전이 창에서 상자 체인+2차 피팅으로 코리도 선을 물고, **락
프레임만 밴드 대체 / 실패 프레임은 기존 경로 통과(폴백) / 창 밖 전 구간 무변경.**

- 파일: lane_detector.py(`_sw_corridor_bands` + process 창 블록), lane_node.py(sw_*)
- 진입 창(440f≈22s, 링 전체 커버): run51/53/54 개구부 포함 안정 — run50 의 7s 창
  만료 이탈을 고침. 탈출 창(400f): 점선 꼬리 추적(run53 병합 표류 대응) + 흰 인계
  종결자(wsteady) + **탈출 개방 구간**(아래).
- **탈출 개방 구간 (07-12 오후, run54 이탈 대응)**: 발화 직후 차는 링 접선 방향이라
  코리도가 좌향으로 들어와 +1 방향 게이트가 코리도 자체를 기각 → 맹목 표류.
  `sw_exit_open_frames`(30≈1.5s) 동안 방향 게이트 해제 + 전체 높이 시딩 + |기욺|
  상한(`sw_open_max_lean_px` 90)으로 정지선 대각선(실측 102~148px)만 기각.
  오프라인 검증: run54 실패 프레임 EX+0.1~0.8 전부 락, run53 은 락 시작이

## 2. 하드웨어 상태 (심각)

- **배터리 팩 수명 종료 중**: 만충 8.26V 에서도 고속 밴드 25s 연속 부하에 브라운아웃
  (run55, RA+25 보드 리부팅). 런별 속도 널뜀의 근원. **교체 팩이 모든 것의 전제.**
- **브라운아웃 리부팅 징후** = 노드 죽고 차는 계속 감 → **즉시 ESC OFF** (PWM 칩이
  마지막 명령 유지 — 소프트웨어로 못 세움). 리부팅 후 /tmp 스크립트 재배포 필수
  (launch 로그는 이제 ~/auto_race.log 라 생존). 급정전 시 ~/frames 는 페이지 캐시
  미플러시로 0바이트 전손 가능 — 텔레메트리(fsync)만 믿을 것.
- 동글 사망 반복 → 뽑고 3s 재삽입, 안 되면 reboot. i2c errno=16 → ESC OFF 후 재시도.
