# D-Racer 자율주행 디버깅 매뉴얼 (에이전트 필독)

이 문서는 실주행 디버깅에서 반복적으로 확인된 사실을 정리한 것이다.
**추측하지 말고 이 문서에 적힌 실측 근거를 우선하라.** 여기 적힌 "건드리지 말 것"은
과거에 시간을 크게 낭비했거나 상황을 악화시킨 항목이다.

## 0. 황금 규칙 (반드시 지킬 것)

1. **한 번에 하나만 바꾼다.** 파라미터 여러 개를 동시에 바꾸면 어떤 게 효과였는지 알 수 없다.
2. **실측 근거 없는 수치 변경 금지.** 모든 수치는 텔레메트리(`~/telemetry.jsonl`)의
   실측값과 대조해서 정한다. "느낌"으로 바꾸지 않는다.
3. **주행 코드를 만지기 전에 ESC를 끄고 손으로 밀며 검증한다.** `ros2 launch`는 실제로
   차를 움직인다.
4. **"멈춰" 지시가 오면 다른 무엇보다 먼저 `bash /tmp/stop.sh`.**
5. **증상을 하드웨어/소프트웨어로 먼저 분류한다.** 아래 §4의 "보드가 죽는 3가지"를
   착각하면 엉뚱한 코드를 몇 시간 뒤진다.

> **절대 손대지 않는다.** 표에 없는 증상이면 §0의 우선순위 절차부터 다시 밟는다.

### 부호(±) 민감 파라미터 — 반드시 의미를 지킬 것

이 값들은 **부호가 방향을 결정**한다. 크기만 바꾸고 **부호는 절대 뒤집지 마라**
(과거 부호 반전으로 링 바깥 이탈 버그 발생 — §1).

- `circle_steer_bias` : **음수 = 링 안쪽**, 양수 = 바깥쪽. 현재 **0** (재도입 시 음수만).
- `entry_steer_bias`  : **음수 = 좌조향 피드포워드**. 현재 **-0.15** (진입 락 중,
  실제 해제는 fork 재수렴으로 1.77s).
- `exit_steer_bias`   : **양수 = 탈출측(우)**. 현재 **+0.18** (탈출 락 2.5s, 소실 프레임 한정).
- `steer` 최종부호는 `steer_scale`(-1 반전)과 엮여 있다. bias 부호만 따로 뒤집으면
  둘의 정합이 깨진다. **bias는 크기만 조정.**
- 층위 요약 (07-11 확정, 문서화만 — 코드 부호 리팩터링 금지): **1층 최종 steer 값은
  >0.26 = 좌.** 2층 바이어스/오프셋은 **음수 = 좌/안쪽, 양수 = 우/바깥** 이고,
  변환식 `steer += turn_direction(-1) × bias` 가 두 층을 잇는다.

### 이 문제(흰→노랑/교차로 오인식) 작업 시 절대 하지 말 것

- `_lane_center` / `compute_angle` 등 **검출·분류 함수 내부를 고치지 않는다.**
- HSV 색범위(`white_low/high`, `yellow_low/high`)를 **추측으로 바꾸지 않는다**
  (바꾸려면 `hsv_filter.py`로 실제 영상에서 재측정 후에만).
- ROI(`roi_top`, `lookahead`) 구조를 바꾸지 않는다 — 이건 로직 변경에 준한다.
- 새 판정식/새 검출 방식을 도입하지 않는다. **위 표의 파라미터만.**

## 1. 절대 건드리지 말 것 / 되돌리지 말 것

- **`yaw_gate_min = 4.2`** — 링 안의 '진짜 실선'을 가짜 정지선으로 오인해 조기 탈출하던
  문제를 실전에서 차단한 값(auto6 + run20/25 로 재검증). 영상 한 장으로는 진짜/가짜 구분이
  원리적으로 불가능하다. **근거 없이 낮추지 말 것.** 단 07-12 실측 주석: 가짜 군집 최대
  3.11 / 진짜 탈출선 4.12~4.28 — 진짜 쪽 여유가 2%뿐이라 속도가 빨라지면 진짜 선을 막는다.
  재배치 후보 3.6(양쪽 ±12%)이 사용자에게 제안된 상태 — **결정은 사용자 몫, 임의 변경 금지.**
- **`circle_steer_bias` 부호** — 음수 = 링 안쪽. steer_scale을 -1로 뒤집을 때 편향 부호를
  같이 안 뒤집어서 링 바깥으로 이탈하던 버그가 있었다. 부호를 다시 뒤집지 말 것.
  (07-11 저녁 확정: 0 — 링은 순수 PID 로 완주 입증(run20). 재도입 시에도 음수만, -0.05 부터.)
- **yaw(누적 조향각) 위치창을 새로 만들지 말 것** — yaw = 조향편차×시간이라 런별 속도
  차이에 창 위치가 통째로 밀린다 (07-12 에 merge yaw 창 [2.45,3.6] 을 이 이유로 폐기,
  이벤트 기반으로 대체). 위치 조건이 필요하면 이벤트(전이) + 시간 하한 조합을 먼저 검토.
- **i2c-3 버스가 잠기면 차가 소프트웨어로 안 멈춘다** (07-11 오후 실사고) — PWM 칩이
  마지막 스로틀을 유지하고, ESC 전원이라 보드를 꺼도 계속 돈다.
  **비상정지 = ESC 스위치 OFF 뿐이다.** "멈춰" 후에도 차가 굴러가면 즉시 ESC 부터.
- **`center_jump_max_ratio` 를 0.10 위로 올리지 말 것** — 0.15 는 분기 목 쐐기 당김
  (프레임당 0.05~0.08)이 상한 아래로 통과해 방어가 뚫린다 (run11 실증).
- **`crossline` 검출은 HoughLinesP 기반이다.** 컬럼평균 최소제곱 피팅으로 되돌리지 말 것.
  정지선이 세로 차선과 T자로 붙어 한 연결성분이 되면 최소제곱이 무너진다(inlier 0.05).
- **`yellow_heading_gain = 0`** — 헤딩 보정 a_h(전체 피팅)가 ±2.5로 널뛰어 요동을 유발했다.
  0이 아닌 값으로 켜지 말 것.
- **i2c 노드를 `kill -9`로 죽이지 말 것.** 전송 중이면 i2c-3 버스가 잠긴다.
  반드시 SIGTERM: `pkill -TERM -f 'install/.*/lib/'`.
- **주행 중 UART를 꽂아두지 말 것.** 케이블이 트랙 한 바퀴를 못 버틴다. 당겨지면서 전원
  커넥터까지 흔들려 `bootreason=cold` 재부팅을 유발한 적이 있다(커널 문제 아님).
  텔레메트리는 `fsync`로 남으니 UART 없이 주행해도 데이터는 안전하다.

## 2. 빌드 / 실행

```bash
cd /home/topst/SEAME_2026_Hackathon-clone/D-Racer-Kit
source /opt/ros/humble/setup.bash
source install/setup.bash
# symlink-install 아님 → 파이썬 파일 수정해도 반드시 리빌드
colcon build --packages-select perception decision
ros2 launch decision auto_race.launch.py course:=in race_dir:=left
```

- **symlink-install이 아니다.** 코드를 고쳤으면 반드시 `colcon build` 후 다시 실행.
  안 그러면 옛 코드가 돈다. "고쳤는데 안 바뀐다" 증상의 1순위 원인.
- 대시보드: `http://<보드IP>:5000`
- 보드 접속: 핫스팟 바뀌면 IP도 바뀐다. MAC `fc:22:1c:40:62:96`으로 찾는다.
  ```bash
  ip neigh | grep -i "fc:22:1c"
  ```

## 3.5. 트랙 구간 지도 + 구간별 알려진 문제 (도면 기준)

트랙은 흰색 차선 외곽 루프 + 중앙 노란색 회전교차로(RA) 구조다. 진행 순서와 각 구간에서
자주 나는 오인식, 그리고 **건드려도 되는 파라미터만** 아래에 매핑한다.
구간을 착각하면 엉뚱한 파라미터를 만진다. 텔레메트리의 `st`(상태)로 현재 구간을 확인하라.

| 구간 | 차선 | 알려진 문제 | 건드릴 것 (로직 금지) |
|---|---|---|---|
| 출발/체크무늬(FINISH) | 흰색 | 출발 직후 원거리 노란 마킹 조기 래치 | `follow_yellow_ratio 0.03`, `ra_min_drive 7.5s` |
| 흰→노랑 **전환부(인코스)** | 흰→노랑 | **핵심 문제 §3.6 참조** | `follow_yellow_ratio`, `yellow_drive` |
| RA 진입 | 노란색 | 진입 직후 급좌회전 언더스티어 + **개구부 nl=2 중점 직진 이탈 (run27, 수정안 승인 대기)** | `entry_steer_bias -0.15`, `entry_lock 3.0s` (실해제 1.77s — fork 재수렴) |
| RA 원형 순환 | 노란색 | 링 안 진짜 실선을 정지선으로 오인해 조기 탈출 / 차선 소실 시 직진 이탈 | `yaw_gate_min 4.2` (§1 — 임의 변경 금지), `ra_blind_bias -0.15` |
| RA 합류부 (접근로 접선, 점선 개구부) | 노란색 점선 | 피팅이 접근로 직선을 잡아 '직진' 오답 / 재획득 단선을 좌측으로 오인 | `reacq_arm_s 5.0` (0→1 재획득=우측 규칙, 07-12 배포·미검증) |
| RA 탈출 | 노랑→흰 | 탈출로 갈아타며 차선 소실 → 우측 표류 / 탈출 직후 오버스티어 (run25) | `gate_sustain 0.25s`, `exit_gates 1`, `exit_steer_bias +0.18`/`2.5s` (소실 한정) |
| 우측 S커브(타이트) | 흰색 | 바깥 차선만 보이고 화면 왼쪽으로 넘어와 좌/우 오인 → 조향 반대 | §3.6 수기 보정만. 로직 금지 |
| 좌측 빨간 구간 | 흰색+빨강 | RA 통과 전엔 red 무시 | `red_slow_ratio 0.05` |

## 3.6. 흰→노랑 전환 & 교차로 오인식 (07-11 오후: **해결됨** — 기록 보존용)

> ★ 이 문제는 07-11 오후 run7~15 로 원인 확정 + 해결됐다: 원인은 문턱이 아니라
> ① 분기 목에서 점선+실선이 nl=2 로 잡혀 중점이 조향을 쐐기로 끄는 것 (run9 EMA 역산),
> ② 안쪽 실선 소실 후 단선 추종의 방향 상실. 해결 = 연속성 가드(`center_jump_max_ratio
> 0.10`) + 진입 병합 1L 모드(안쪽 실선 소실 순간부터 80프레임 ≈4초, 전체 평균을
> 바깥선 하나로 — run17 에서 2초→4초 확대). 이후 12+ 연속 진입 성공.
> **현재 최우선 전선은 RA 진입 개구부 통과다 (§3 참고).** 아래는 당시 기록.

**현상 (사용자 확인):**
- 흰색 차선 주행 중 **노란색을 `follow_yellow_ratio`(현재 0.03) 이상 인식하면 흰색을 무시하고
  노란색을 따라가도록** 되어 있다. 그런데 이 전환 지점에서 오인식이 난다.
- 교차로(RA)에서도 오인식한다: 안쪽 차선이 안 보이고 **바깥 차선만 ROI에 들어올 때**,
  차가 우측 차선을 좌측으로 오인해 우측으로 가려 한다. 실제로는 **좌측으로 가는 게 정답**.

**대응 원칙 — 로직을 고치지 말 것 (§0-0).** 검출/분류 알고리즘을 새로 짜지 않는다.
아래 두 지점만 건드린다.

1. **전환 문턱 조정 (파라미터만):**
   - `follow_yellow_ratio`(현재 0.03)가 **너무 낮으면** 흰 구간에서 원거리 노란 마킹에
     조기 전환(오인식). **너무 높으면** 전환부에서 노란선을 늦게 잡아 흰선을 붙들다 튄다.
   - 조정 시 **한 번에 0.005씩**, 텔레메트리 `yr`(yellow ratio)로 전환 순간의 실측
     `yr` 분포를 보고 정한다. "느낌"으로 바꾸지 말 것.
   - 흰 구간 실측 `yr` 최대값보다 확실히 위, 진짜 노란 진입부 `yr`보다 아래에 문턱을 둔다.

2. **바깥 차선만 보일 때 수기 보정 (로직 아님, 규칙 적용):**
   - 상황 조건: "안쪽 차선 미검출 + 바깥(우측) 차선만 ROI에 존재".
   - 이때는 **분류 결과를 신뢰하지 말고 좌측 조향으로 보정**한다(사용자가 노가다로 확정).
   - 이 보정은 **해당 상황 조건에서만** 적용한다. 조건을 넓히거나 판정 알고리즘 자체를
     바꾸지 않는다. 조건/보정 방향은 사용자가 정한 값을 그대로 쓴다.

**검증:** 두 변경 모두 텔레메트리로 before/after를 비교한다.
- 전환부: 전환 시점의 `yr`, `nl`(차선 수), `off`(오프셋) 튐 여부
- 교차로: 바깥 차선만 보이는 프레임에서 `steering` 부호가 좌측으로 나오는지
- **개입 구간은 분석에서 제외**(§6).

## 4. 보드가 죽는 3가지 방식 (혼동 금지)

증상별로 원인과 복구가 완전히 다르다. 착각하면 엉뚱한 코드를 뒤진다.

### ① Wi-Fi 드라이버 사망 — 가장 흔하고, 보드는 멀쩡함
- 로그: `__cfg80211_connect_result`, `RTW: ERROR [HALMAC]...send h2c pkt fail`
- **ROS 노드는 계속 돈다. SSH/대시보드만 끊긴다. 주행엔 무해.**
- 복구: 시리얼 콘솔에서 `sudo nmcli device wifi connect "<SSID>" password "<PW>"`.
  `device was removed`면 동글 뽑았다 3초 뒤 재삽입. 최후엔 `sudo reboot`.

### ② i2c-3 버스 잠김 — 전원 완전 차단만이 답
- 로그: 빈 주소(0x10, 0x50)까지 `errno=16 (busy)`, `failed to get i2c-3`
- `control_node`(PCA9685 0x40), `battery_node`(INA219 0x42)가 못 뜬다.
- **재부팅으로 안 풀린다.** ESC 끄기 → 보드 전원 뽑기 → **10초 대기**(커패시터 방전) →
  ESC 켜고 → 보드 전원.
- ⚠ **주행 중 잠기면 차가 소프트웨어로 안 멈춘다** (07-11 오후 실사고): 중립 PWM 을
  쓸 수 없어 PWM 칩이 마지막 스로틀을 유지하고, ESC 전원이라 보드를 꺼도 계속 돈다.
  **즉시 ESC 스위치 OFF.** 빈 주소 판별은 smbus 로: 정상이면 빈 주소가 errno=11.

### ③ 진짜 커널 패닉 / 워치독 — 원인 미상
- `bootreason`이 panic/watchdog. 커널 메시지가 안 남음(`CONFIG_PSTORE` 꺼짐).
- 패닉 트레이스를 잡으려면 전원 넣기 **전에** picocom을 로그와 함께:
  ```bash
  sudo picocom -b 115200 /dev/ttyUSB0 -g ~/serial.log
  ```
  `-g` 없으면 기록 안 남는다. USB를 만졌으면 picocom을 다시 띄운다.
- 시리얼 캡처는 **차를 세워두고 auto_race를 방치**해서 잡는다(주행 데이터 위험 회피).

### 부수: eMMC 손상
- 루트가 `norecovery`(ext4 저널 꺼짐). 하드 락업 중 전원 끊으면 initramfs `#`로 떨어진다.
- 복구(USB에 `e2fsck.static` 있음):
  ```bash
  mkdir -p /u && mount -t vfat /dev/sda1 /u
  cp /u/e2fsck.static /tmp/e2fsck && chmod +x /tmp/e2fsck
  /tmp/e2fsck -fy /dev/mmcblk0p4
  sync && reboot -f
  ```

## 5. 배터리 (실패 1순위 원인)

- **충전 중 단자 전압은 잔량이 아니다.** 충전기 떼고 5분 안정화 후 무부하 전압을 재라.
- **팩이 손상돼 있다** (07-10 밤 6.6V 심방전 → 내부저항 증가): 무부하 8.2V 여도 부하에서
  깊게 처진다. run16(8.07V 출발 불가)/run18(8.2V 손밀기) 실측. 이 때문에 07-11 저녁
  스로틀 전체 +0.01 (kick 0.23 / drive 0.19 / yellow 0.17 / slow 0.165, §8). **내리지 말 것.**
- **저전압 보호를 켜라**: `undervolt_slow_v`, `undervolt_stop_v`가 0.0이면 경고 없이 방전된다.
- `battery_node`(INA219 0x42)를 근거 없이 끄지 말 것. 끄면 전압을 못 본다.
  (현재 `use_battery=false` — 복원 대기 항목, HANDOFF §F.)
- 수동 전압 확인 (노드 없이): bus3 addr 0x42 reg 0x02,
  `v = ((raw&0xff)<<8 | raw>>8) >> 3 × 0.004` (워드 바이트 스왑 필요).

## 7. 핵심 알고리즘 메모 (이해용 — 수정 대상 아님)

> ⚠️ **이 섹션은 "왜 이렇게 짰는지" 이해하기 위한 것이지, 고치라는 뜻이 아니다.**
> 여기 적힌 알고리즘들은 이미 실주행에서 검증됐다. 증상이 보여도 알고리즘을 다시
> 짜지 말고, §8 파라미터 또는 §7-특수상황의 수기 보정으로만 대응한다.
> 알고리즘 변경은 §0-0에 따라 사용자 승인 후에만.

### 특수 상황 수기 보정 (로직 대신 이것만 적용)

- **단선 좌/우 오인 (교차로/급커브에서 우측 차선이 화면 왼쪽으로 넘어올 때)**:
  검출기가 우측 차선을 좌측으로 오분류해 우조향으로 튄다. **분류 알고리즘을 고치지 말 것.**
  사용자가 노가다로 확정한 규칙 — "이 상황(안 차선 안 보이고 바깥 차선만 보임)에서는
  **좌측으로 가는 것이 정답**" — 을 그대로 적용한다. 사용자가 잡아준 상황 조건과
  보정 방향을 벗어나 임의로 판정 로직을 바꾸지 않는다.


### 정지선(crossline) 검출 — `_detect_crossline`
- 세 겹 버그를 고친 결과다: (1) 최소제곱→HoughLinesP 교체, (2) a_h를 스캔창 같은 행에서만
  계산 후 후보 선분 지우고 재계산, (3) BEV가 각도 비보존(`crossline_bev_aspect = 1.91` 실측)
  이라 지면 좌표로 되돌린 뒤 직교 판정.
- 판정: 차선 방향 (a_h,1)·선분 (1,slope)를 지면으로 → (a_h/r,1),(1/r,slope),
  `|cos(사잇각)| <= sin(20°)`면 정지선. 차선 오검출 시 slope=1/a_h라 cos=1(평행)→어떤
  헤딩에서도 통과 불가. **결과: 진짜 정지선 100%, 노란 차선 오검출 0%.**

### RA 탈출 조건 (적용됨 — run20/25 로 검증)
- 탈출 게이트 = 시간(`min_loop 17s`) AND yaw(`yaw_gate_min 4.2`) AND 정지선 엣지
  (블랭크 17s + 재무장 0.5s + 지속 0.25s). 백업 표결(yaw_lap 9.0)과 `max_loop 30s`
  failsafe 는 별도 유지.
- `yaw_proxy` 는 최종 명령 조향의 한방향 적분 (`defl = -turn_direction×(steer−center)`,
  defl>0 만 누적, RA 진입 시 리셋). **바이어스 포함 최종값을 적분하므로 순수 헤딩이
  아니고, 속도가 느리면 같은 위치에서 값이 커진다** — 위치 프록시로 쓸 때의 함정 (§1).
- 실측 눈금: 가짜 실선 군집 1.5~3.11 / 진짜 탈출선 4.12~4.28 / 한 바퀴 ≈4.15~4.3.

### RA 조향 체계 (07-12 재설계 후 현행)
```
진입 락 (~1.77s, fork 재수렴 해제)  = PID + entry_steer_bias(-0.15)
순항                               = 순수 PID (circle_steer_bias 0)
소실 프레임                         = + ra_blind_bias(-0.15) 좌호
RA+5s 후 nl 0→1 재획득             = 그 선을 우측(바깥) 경계로 분류 (reacq_arm_s)
게이트 탈출 후 2.5s 소실 프레임      = + exit_steer_bias(+0.18) 우호
```
- yaw 위치창 + 하드 오버라이드(구 merge_yaw_lo/hi, merge_steer_bias)는 07-12 폐기.
  파라미터 선언만 남아 있고 **미사용** — 다시 배선하지 말 것 (§1 yaw 창 금지 참조).
- 0→1 만 규칙에 쓰는 근거 (RA 에피소드 8개 실측): 5s 이후 0→1 은 합류부에서만 발생
  (오발 0건), 2→1 은 순항 중 일상적(3~18회/랩)이라 넣으면 순항이 깨진다.

### 단선 좌/우 오인
- 급좌회전에서 오른쪽 바깥 차선이 BEV상 화면 왼쪽으로 넘어오면 왼쪽 차선으로 오분류 →
  조향이 반대로 튄다. `_yellow_dash_cx` 힌트가 있는데 왜 안 먹는지 확인 필요.
- 직전 중심 연속성 기반 분류가 추가돼 있음(프레임마다 뒤집히던 문제 대응). 단 연속성도
  뚫린다: run26 에서 2→1 전이 때 남은 우측선을 좌측으로 오인 → off +1.00 (한계 인지).
- RA 안에서는 위 "0→1 재획득=우측" 규칙이 담당. RA 밖은 여전히 수기 보정 원칙.

  올리는 방향만, `yr` 실측 근거 필수. `yellow_slow_ratio` 와 항상 동기 유지할 것.
- 17초는 자율 랩타임 실측이 쌓이면 재확정(진짜 탈출 ~20초+ 대비 여유 3초뿐).
- 라이브로 바꾼 값은 검증되면 소스 기본값으로 승격한다.

## 9. 디버깅 시 에이전트 행동 지침 요약

- 증상 보고를 받으면: **하드웨어(§4·§5) 먼저 배제 → 그다음 코드.**
- "고쳤는데 안 바뀐다" → **colcon build 안 했는지 먼저 확인**(symlink-install 아님).
- 파라미터를 제안할 때는 반드시 **어느 텔레메트리 필드로 검증할지** 함께 제시한다.
- §1의 금지 항목을 되돌리는 제안은 하지 않는다. 꼭 필요하면 근거(실측)를 먼저 요구한다.
- 두 번 같은 시도가 실패하면 증상을 다시 분류하고 접근을 바꾼다(무한 패치 금지).

## 10. 외부 레퍼런스 채택 판단 (조사 완료 — 무엇을 쓰고 무엇을 버릴지)

> 아래는 GitHub/웹의 공개 자료를 우리 프로젝트 원칙("로직 재작성 금지, 파라미터만")에
> 비춰 평가한 결과다. **결론: 대부분 채택하지 않는다.** 이미 해결된 것을 다시 건드리거나
> 접근법이 다른 것을 도입하면 검증된 동작이 깨진다. 아래 판단을 뒤집으려면 실측 근거를
> 먼저 제시할 것.

### ① OpenCV 차선 중심 오프셋 계산 — ❌ 채택 안 함 (이미 해결)
- 공개 예제(CarND/Advanced-Lane-Lines 계열)는 **다항식 피팅 + 곡률 + 차선중심 오프셋**
  방식이다. 우리는 이미 BEV + `crossline_bev_aspect=1.91` 보정으로 off ±0.1 안정 구간을
  확보했고(§3), 정지선 검출도 HoughLinesP로 재작성해 100%/오검출0%를 달성했다(§7).
- **그들의 다항식 피팅으로 되돌리면 §1에서 금지한 컬럼평균 붕괴 문제가 재발**한다.
  차선이 정지선과 T자로 붙으면 최소제곱 inlier가 0.05까지 떨어진다.
- 참고할 부분이 딱 하나 있다면 **오프셋→조향의 PD 게인 튜닝 관점**뿐인데, 이것도
  우리는 파라미터(steer gain)로 이미 조정 중이다. 새 공식 도입 불필요.
- 판단: **우리 오프셋 로직은 검증됨. 새 알고리즘 이식 금지. 조정은 게인 파라미터로만.**
  — 근거조사: web/github (CarND 계열 6개 리포 확인). 서명: Kiro, 2026-07-11

### ② Duckietown 차선 제어 루프 — ❌ 채택 안 함 (접근법 상이)
- Duckietown LF 데모는 **색상 검출(color detection) 기반**이고, 실행 가이드 성격이라
  디버깅 지식이 거의 없다. 우리 파이프라인(BEV + Hough + 상태머신 + steer bias)과
  아키텍처가 달라 코드 재사용 불가.
- 유일하게 취할 교훈: "색조명/반사가 카메라를 교란한다" — 이건 우리도 이미 겪는 문제라
  **트랙 조명 관리** 운영 팁으로만 유효. 코드 변경 사항 아님.
- 판단: **참고 자료로만. 코드/제어 루프에 이식하지 말 것.**
  — 근거조사: docs.duckietown.com LF 매뉴얼. 서명: Kiro, 2026-07-11

### ③ Raspberry Pi OpenCV FPS 최적화(멀티스레딩) — △ 부분 채택 (단, 로직 아님)
- 검증된 개념(pyimagesearch, Qengineering/Multithread-Camera): **캡처 스레드를 분리해
  I/O 대기로 메인 루프가 막히지 않게 하고, 항상 "가장 최신 프레임"만 처리**한다.
  이것이 왜 우리에게 중요한가 → **제어는 지연에 민감**하다. 프레임이 쌓여 오래된 영상으로
  조향하면 링/코너에서 뒤늦게 반응해 이탈한다(우리가 겪는 요동의 일부일 수 있음).
- **하지만 카메라 파이프라인 변경은 "로직"이다.** 대회 중 캡처 구조를 바꾸는 것은 위험.
  camera_node가 죽으면 주행 전체가 조용히 멈춘다(§respawn 설치된 이유).
- **안전하게 취할 수 있는 것(파라미터/설정 레벨, 로직 아님):**
  1. **해상도 하향** — 320x240 이하에서 실시간 가능(karstenBehrendt 실측: 그 이상은 <10fps).
     우리 카메라 설정에서 처리 해상도를 낮추면 프레임 지연 감소.
  2. **오래된 프레임 드롭** — 구독 큐 depth를 1로(최신만). ROS2 QoS `depth=1` +
     `best_effort`로 조정 가능. **이건 QoS 파라미터라 로직 재작성이 아니다.**
  3. **프레임 스로틀** — 처리 못 따라가면 입력 FPS를 낮춰 밀림 방지.
- **하지 말 것:** 스레드 캡처 클래스 신규 작성, camera_node 내부 재구조화(= 로직 변경).
- 판단: **QoS depth=1 / 해상도 하향만 검토. 스레딩 재작성은 금지.**
  적용 시 반드시 텔레메트리로 프레임 지연·off 요동 감소를 검증할 것.
  — 근거조사: pyimagesearch, Qengineering, karstenBehrendt/raspi_opencv_performance.
  서명: Kiro, 2026-07-11
